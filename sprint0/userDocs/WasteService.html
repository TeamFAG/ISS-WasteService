Meaning of nouns and verbs

NOUNS:

Abstract Entities

WasteService:
Software that allows a Waste truck to deposit its load in the proper container.
Service Area:
A rectuangular area that includes an INDOOR port, a PlasticBox container and a GlassBox container. It's free from any internal obstacles;
INDOOR:
A port to the Service Area. Its formal position and usage will be later defined;
PlasticBox:
A container devoted to store objects made of plastic up to a limited capacity;
GlassBox:
A container devoted to store objects made of glass up to a limited capacity;
HOME:
The initial and default location of the Robot. It is found inside the Service Area in a formal location later defined;
Driver:
The driver of the Waste truck which uses a smart device to interact with the WasteService;
ServiceManager:
A human being that supervises the state of the WasteService;
RaspberryPi:
An hardware device that can interact with components of the WasteService;

Formalized Entities

DDR Robot:
Differential Drive Robot, that can be virtual or real. Capable of moving in the environment by executing simple move commands. It must be capable of moving objects as a transpot trolley with a squared form of fixed side length equal to RD.
The customer provides the software that further defines its capabilities: it.unibo.basicRobot22;
The following is a visualization of the complete Qak model that the customer gives us:

The main component of the architecture is the basicrobot actor that encapsulates the robot's (virtual or real) logic.
In the Requirement Analysis, we will use only this actor to represent the DDR robot, as it describes its basic behavior.

The pathexec actor simplifies the interaction with the DDR robot by asking for a path and not a single command.
We will explore the possibilities of its use in the next Sprint documents.

The actor receives dispatch messages called cmd. The MOVE argument in these messages identifies a type of movement based on the aril language.
//MOVE =  w | a | s | d | h 
    
Dispatch cmd : cmd(MOVE)
We also have a step Request (with its connected Replies) to dictate the timing of movements.
Request step : step(TIME)
Reply stepdone : stepdone(V)
Reply stepfail : stepfail(DT, REASON)
Finally, an Event is emitted whenever the robot collides with something. 
Event info : info(ARG)
Sonar:
A device that measures a distance. It's connected to the RaspberryPi.
The customer provides us the following softwares: it.unibo.radarSystem22.domain and unibo.sonarqak22.
The QActor from unibo.sonarqak22 exposes two dispatches and an event. The commands sonaractive and sonardeactive are used to enable and disable the sonar, insted the sonar event is used to send the distance to the next actor in the pipe. 
Dispatch sonaractive : info(ARG)
Dispatch sonardeactive : info(ARG)
Event sonar : distance(V)

The dataCleaner actor takes care of cleaning 'wrong' data and the distanceFilter actor handles the filtering, sending the data only if the detected distance is less than a fixed value.
Led:
A light that can be switched on or off. It's connected to the RaspberryPi;
The customer provides us the code Led25GpioTurnOn.sh and Led25GpioTurnOff.sh: these bash scripts simply turn on and off the led connected to a Raspberry;
Waste truck:
A transport vehicle that brings waste material to the INDOOR port;
WasteServiceStatusGUI:
User interactive component used by the ServiceManager to observe the status of the WasteService:
  • the current state of the transport trolley and its position in the room;
  • the current weigth of the material stored in the two waste-containers;
  • the current state of the Led.
StoreRequest:
The request sent by the WasteTruck; 
Request storeRequest : storeRequest(MATERIAL, QUANTITY)
LoadAccepted:
The reply sent by the WasteService if the load can be accepted; 
Reply loadAccepted : loadAccepted(_)
LoadRejected:
The reply sent by the WasteService if the load can't be accepted; 
Reply loadRejected : loadRejected(_)
MAXPB:
Fixed max kg of material which can be stored in the PlasticBox;
For the following definitions we assume that MAXPB is 100KG, this will likely change in the future. 
val MAXPB : Double = 100.0
MAXGB:
Fixed max kg of material which can be stored in the GlassBox;
For the following definitions we assume that MAXGB is 100KG, this will likely change in the future. 
val MAXGB : Double = 100.0
DLIMIT:
Fixed distance compared to the distance measured by the sonar, if the latter is less than this value the Robot must be stopped; 
val DLIMIT : Float

VERBS:
  • Start the robot: start the robot from the position it is currently at. This operation is always managed by the WasteService;
  • Stop the robot: stops the robot in the position it is currently at. This operation is always managed by the WasteService;
  • Pick up load: the robot takes the waste load from the Waste Truck;
  • Deposit load: the robot unloads the waste in the appropriate container;

Definition of a formal model of the requirements

For the following formalizations we used the Qak modeling language because the customer already provides the formalized models of some of the entities.
It's a fast and easy tool to model executable and testable architectures

Driver Request Requirement

The first model explains the interaction between the driver/WasteTruck and the WasteService.
The driver, the WasteTruck and the Smart Device used for the request at this moment have been joined in a single Actor, called wastetruck.
The other Actor present in this first model is wasteservice, who takes care of managing the whole system. This was done to simplify the visualization and understanding of the model. The complete model can be seen here: model_request.qak
The interaction is based on a request sent by the driver with the Smart Device, as described in the requirements.
Its payload contains both the Material type and quantity inside the WasteTruck. 
Request storeRequest : storeRequest(MATERIAL, QUANTITY)
Reply loadAccepted : loadAccepted(_)
Reply loadRejected : loadRejected(_)
Once the Request is received by the wasteservice, based on the type of material and the current free space in the respective container, we choose to accept or reject the storing of the load. 
onMsg(storeRequest : storeRequest(MATERIAL, QUANTITY)) {
    [#
	    currentMaterial = ws.Material.valueOf(payloadArg(0))
	    currentQuantity = payloadArg(1).toDouble()
    #]
    		
    if [# currentMaterial == ws.Material.GLASS && occupiedGlass + currentQuantity < ws.WasteServiceConstants.MAXGB || 
                currentMaterial == ws.Material.PLASTIC && occupiedPlastic + currentQuantity < ws.WasteServiceConstants.MAXPB #] {
	    replyTo storeRequest with loadAccepted : loadAccepted(_)
    } else {
	    replyTo storeRequest with loadRejected : loadRejected(_)
    }
}
At this point the wastetruck receives the response and behaves accordingly. This behaviour will be later defined in the problem analysis. 
Transition t1 whenReply loadAccepted -> handleAccepted
              whenReply loadRejected -> handleRejected
The driver/WasteTruck should then leave as soon as possible, but who communicates to the driver that he can leave the indoor? We have 2 options:
  • The trasport trolley directly notifies the driver (through the smart device) that the load has been picked up.
  • The WasteService notifies the driver (through the smart device) that the load has been picked up. The consequence of this particular solution is that the trolley will have to notify the WasteService after completing the pickup action.
The choice between thiese to strategies will be discussed in the Problem Analysis of the next Sprint.

Deposit Requirement

The next model describes the interaction between the WasteService and the trolley.
As of now, this model does not take into consideration the situation where the trolley is busy while receiving a message.
That situation will be handled in the next Sprint.
Moreover, for semplicity's sake, the whole context only simulates one interaction. The complete model can be seen here: model_deposit.qak
The message type used for both of the actors is dispatch.
There is no need for a Request/Reply in this interaction, as the trolley and wasteservice only need to be notified of the situation and react accordingly. 
Dispatch notifyDeposit : notifyDeposit(MATERIAL, QUANTITY)
Dispatch updateWasteService : updateWasteService(MATERIAL, QUANTITY)
wasteservice Actor
The state sendDeposit is used by the wasteservice actor to notify the trolley of an accepted deposit request.
With this notification he also comunicates the type of material to be deposited and it's quantity. 
State sendDeposit {

    forward transporttrolley -m notifyDeposit : notifyDeposit($MaterialType, $Quantity)
}
Goto idle
When the trolley completes a storing task, it sends an updateWasteService dispatch to the wasteservice actor with the necessary information to update the state of the two containers.

System Architecture

We now join all the previous models into a complete system architecture.
The model can be seen here: architecture.qak
During the creation process of this final model we have made a few changes to the single actors:
  1. The basicrobot22 project has now been integrated as a used component by the trasporttrolley via docker image.
  2. All the necessary actors now emit update Events to be used by the GUI.
    These are some examples:
    emit updatePosition : updatePosition($Pos)
emit updateTrolleyStatus : updateTrolleyStatus($Status)
  3. As a preview to the problems to be handled in the next Sprint, the trolley interaction has been changed to a different logic.
    The update of the current weigth inside the system is now handled by the wasteservice as a consequence of the acceptance of a request.
    This grants us the ability to know the current weigth before the completion of a deposit, allowing a faster and more accurate response to any incoming storeRequest.
