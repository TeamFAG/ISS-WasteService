Importance of clean architecture in relation to the UI

The clean architecture is a way of separating responsibilities and parts of functionality according to their proximity to the application domain.

By the domain, we mean the part of the real world that we model with a program. This is the data transformations that reflect transformations in the real world.
For example, if we updated the name of a product, replacing the old name with the new one is a domain transformation.



In the scheme of the architecture, our GUI component is an external interface and as such it should be detatched from the core business logic of the system.

Desktop Application

With this strategy, the WasteserviceStatusGUI component is an executable that needs to be installed on the target machine.
  1. - The App can be independent from the system (if developed as an external program), however it still will be highly dependent on the OS/platform used to run it.
  2. - If developed as an internal component (part of the qak architecture with a Qak actor/codedActor), it will be too dependent on the system itself, limiting present and future development.
  3. - Running this application using Docker is always more difficult, as visualization of the GUI needs to be handled inside the container.

Web Application

With this strategy, the WasteserviceStatusGUI component runs on an external server as a website.
  1. + The webApp is detached (or independent) from the system.
  2. + The customer provides support for this choice by presenting other functioning projects.
  3. + There are no OS/platform limitations as the only required application is a browser.
  4. - Defining the interaction between a Qak based system and an external server could prove to be challenging.

WasteService to GUI communication handling

One of the core problems of this Sprint is the communication of the system and the GUI as the latter needs to be constantly updated by the system.
Infact, the only porpouse of the GUI is to receive status updates from the service and show them to the manager.
Because of this the only two message types that make logical sense are:
  • Dispatches
  • Events
As we already discussed in Sprint2, Events are sent to all endpoints in the system and represent additional load on the netework.
So Dispatches seem to be the best option.
However, after further analysis the team realised that Dispatches require the knowledge of the receiving end of the connection.
Which means that they introduce a level of dependency that should be avoided.
We will tackle the issue in a separate part of this Sprint document.

Update policy

How do we know when a gui update is needed ?
This particular problem is very similar to the sonar and led analysis of Sprint2, in which we had to monitor the trolley's behavior to update the state of the system.
Consequently, for the same reasons, we decided to use the Observer pattern by implementing CoaP messaging.

Our pre-existing trolleystateobserver actor already observes the state ad position of the trolley.
It also acts as the notifier for the Led state, so reusing it is the best possible choice.
Dispatch updateLed : updateLed(STATE)
Dispatch updateRobotState : updateRobotState(STATE)
However, one required information for the gui is not being observed. The containers' state.
To solve this, we created another observer, wasteserviceobserver that runs in the ctxwasteservice qak context.
This actor receives CoaP messages sent by the wasteservice actor whenever a container is updated by the system (by using updateResource funcitons).
Next, it sends Dispatches to notify of the update of the GUI.
Dispatch updateMaterial : updateMaterial(MATERIAL, QUANTITY)
Who receives these messages ? We introduced a new actor called guiupdater that receives Dispatches from all the update sources (Observers) and consequently uses them to create and ultimately send the final update message.

This was done so that the eventual GUI component will only have to receive one type of message from the system, instead of having to parse each message differently based on its origin.



For now, the only entity that listens to these updates is a CoaP observer built as part of a Test for this particular analysis.
In the project portion of this document we will tackle the issue of the real observer.

Connecting to the Wasteservice system

In the problem analysis section of the document, we defined a CoaP based software structure to collect and send all updates of the system to the GUI.

To finally complete the implementation of this structure, we defined a new object called wsDispatcher the uses the WsConnection class (provided by the customer), to connect to the GUI server and send all new updates received by guiupdater.

Connecting to the Wasteservice system - Solving issues presented while testing

While testing the previous strategy, we found a system-breaking issue involving CoaP based messaging.
Receiveing messages from multiple observers
Apparently, CoaP fails when a QakActor is observing multiple resources.

What happens is that of all the messages sent as part of the updateResource function, The observer only receives messages from one resource, losing all other messages.

In our case, the updates sent by transporttrolley were not received by guiupdater.

A new strategy for the GUI interaction

We then decided to change the whole messaging architecture between the GUI and the Core System.

The first thing that has been done is to remove the two actors introduced earlier (guiupdater and wasteserviceobserver).
CoaP messaging is still used, howerver we moved the observers to the GUI side.
Now there is an observer for every resource of our system:
  • LedObserver: observes the led actor.
  • TrolleyStateObserver: observes the trolleystateobserver actor.
  • TrolleyPositionObserver: observes the trolleystateobserver actor.
    Initially, TrolleyPositionObserver observed the transporttrolley actor as it was the obvious source of data about the position of the trolley, however we found that it wasn't being correctly observed by the GUI CoaP class.
    We fixed this issue by incorporating the information about the position of the trolley into trolleystateobserver
  • WasteServiceObserver: observes the wasteservice actor.
Every time an updateResource for one of the observed actors is called, the new kotlin observers receive the update and use SpringBoot to push it to the live web-page. The consequences of this are:
  • All updates are independent from one another, which means that the GUI is constantly being updated without any delays.
    However, this also means that the network takes on a bigger load as there are more messages.
    For the sake of this project, after further testing, this solution doesn't have any big impacts.
  • Coap doesn't behave strangely as no kotlin class observes more than one actor in the Core System.
  • This strategy makes the GUI component more independent as it can be cahnged, reused and repourposed without affecting the Core System.

Smart Device re-definition

Up to this point, our Smart Device has always been a python GUI script that ran inside the main PC.
However, while starting to develop the wasteservice GUI, we wanted to make the Deposit Request experience closer to the actual real life counterpart.

To do this, we chose to create a mobile app that could connect to the system from anywhere, using Sockets.
By doing so, we simulate the requesting process completely.

As a "mobile framework" we used React Native, which was the better choice for the following reasons:
  • Versatility: The final app can be adapted for Android and iOS.
  • Familiarity: As previously discussed, the team is already familiar with the React family of libraries.
  • Extensive Support: There are many pre-made assets that significantly sped-up the development process.

MQTT and its possible use in our project

Right now, our GUI needs to know the endpoint of the Core System to be able to receive CoaP update messages.
This means that there is still some dependency between the two components, which violates the clean architecture mentioned in the analysis section of the document. How can we completely detach the GUI from the system ? MQTT is messaging protocol designed for IoT messaging that uses the Pub/Sub design pattern.

Messages are sent to/from an entity called message broker that handles the traffic between endpoints
Our GUI interaction would greatly benefit from using this protocol as the core system would only need to know the broker, not the GUI server.

These would be the required steps:
  1. Create a topic on the broker specifically for GUI messages (ex. "guiupdates")
  2. The GUI subscribes to guiupdates so that all messages published on that topic are automatically received.
  3. The system publishes all updates on the topic by sending specific messages to the message broker.
This means that each component involved in the interaction can change and evolve without the need for updates on the opposite endpoint, creating complete independency.

The strategy will be implemented as a future update to the service.

Updates to the distribution of the system

After the final changes to the architecture, we have updated the methods and tasks for a successful deployment.

Core System
In order to run the CoreSystem has been developed a script. This script (run.sh) allows both to launch the system by choosing which context to start, and to automate the execution of all the test plans of the whole system. 
// run.sh usage:

./run.sh [service] [testplan]

wasteservice := wasteservice | trolley | all
wasteservice -> runs the ctxwasteservice, trolley -> runs the ctxtrolley, all -> runs both the ctxwasteservice and the ctxtrolley

testplan := test1 | test2 | test3
test1 -> execute all the tests that are in the Sprint1 project, test2 -> all the tests that are in the Sprint2 project, test3 -> Sprint3 project tests
